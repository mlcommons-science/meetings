## **April 7, 2026 (9.05 pm. ET for Asia-USA)**

### **Present**

### **Meeting Notes**

- [MLC Science WG - 2026/04/08 01:57 BST - Notes by Gemini](https://docs.google.com/document/d/12Kwrj8R6hjLfBjqYBMDeeBBhOu7S5672EVFtAaEWq3Q/edit?usp=sharing)

- **Follow-up on Agentic Research and Collaboration with Sean**

  - Victor Lu confirmed Sean's interest in collaborating on the agentic topic, and Geoffrey Fox committed to following up to secure this alliance.

- **Update on File Formats Research and Lack of Academic Guidance**

  - Victor Lu reported difficulty finding academic guidance or researchers on file formats, confirming a lack of academic research in this area, which suggests the project will be a long-term effort.

  - Only Anna and one other person have signed up for the file format project so far.

  - Geoffrey Fox mentioned Parquet and Lance as relevant file formats.

- **Current Status of File Format Documentation and Interpretation**

  - A UK government resource lists file formats for archival but lacks documentation for application-level semantics within common formats like Parquet or CSV.

  - Agentic AIs require either data descriptions embedded within the agent or self-documenting data schemas for interpretation.

  - Victor Lu was invited to work on the Cloud Data Management Interface (CDMI) within the SNE organization to leverage metadata like file schemas using a hierarchical method, potentially in the API media type header.

- **Discussion on Related Concepts: Data Frame Algebra and Modin**

  - Geoffrey Fox introduced the concept of data frame algebra, established by Modin from Berkeley, as a theoretical framework to systematize and categorize different operators.

- **Ad Hoc Nature of File Identification and Schema Description**

  - File identification tools, like the Linux file utility, operate through an ad hoc process with no current standard or central repository for supporting file definitions.

  - File formats appear to lack related academic courses or research, unlike databases.

  - A starting taxonomy on a wiki page was found, but much of the data is obsolete and requires curation.

- **Recommendation to Document Work and Prioritize Agentic Benchmarking**

  - Geoffrey Fox suggested starting a document to capture file format concepts and discussions.

  - Victor Lu recommended prioritizing agentic benchmarking for the current year, as it is a "hot" topic with existing efforts by Sean to create an actual benchmark.

  - The collaboration strategy is for Sean's agentic working group to focus on principles/benchmarking, with results being "transfer learned" to the science working group for application to scientific domains.

- **Repositories and Registries for File Formats (Pronom and IANA)**

  - Pronom, maintained by TNA in the UK, was identified by Gary Mazzaferro as the only "real resource" for describing file formats.

  - The IANA team is not interested in a file format registry but accepts submissions for MIME types (Parquet and HDF5 were recently added).

  - Agentic systems require data schemas to be self-documenting to prevent the agent's programmer from having to manually define the format and taxonomy.

- **Dynamic Analysis and Feature Set Development (Time Series)**

  - Gary Mazzaferro is compiling a feature set of system features for dynamic analysis (related to time series) from 18 (eventually 25) domains.

  - The list includes approximately 1,600 unique features, consolidating to about 1,100 unique dynamic features.

  - The plan is to seek a grant to publish this information and maintain a website that abstracts dynamics in mathematical terms, providing cross-domain translations.

  - This work provides a standard method for metrics to tie back to algorithms, feature requirements, and workflows.

- **Discussion on the Purpose and Scope of Benchmarks and Taxonomies**

  - TUES DAY questioned the purpose of the benchmark taxonomy (e.g., preventing liability or creating documentation).

  - Gary Mazzaferro described taxonomies as "epistemological constraints on systems" needed to define precisely what is being measured consistently across multiple domains.

  - It was agreed that automation technology, rather than humans, should be used to manage the complexity of standardization.

- **Benchmark Saturation and Data Contamination**

  - Victor Lu defined saturation as models being trained on the benchmark, leading to 100% correct scores.

  - TUES DAY clarified that saturation occurs when metrics hit a ceiling due to data contamination and rapid model scaling, causing a loss of discriminative evaluation over time.

- **Converting Embryonic Documents into Deliverables and the Value of Communication**

  - Geoffrey Fox suggested converting embryonic documents into endorsed 'gems'.

  - Gary Mazzaferro committed to publishing their workflow for developing the dynamics taxonomy on Google Drive for review.

  - Geoffrey Fox proposed compiling contributions from Victor Lu, TUES DAY, Gary Mazzaferro, Armstrong Foundjem, and Gregor von Laszewski into a list.

- **Motivation Behind the Dynamic Feature Taxonomy**

  - The dynamic feature taxonomy aims to define system behaviors based on their dynamics, creating a common, abstract language for modeling and prediction.

  - This is necessary because fields like marketing, supply chain, and military strategy lack a common reference for analysis due to differences in defined metrics and variables across domains.

- **Formulating Precise Deliverables and Next Steps for the Group**

  - Gregor von Laszewski stressed the importance of formulating precise deliverables and defining the consequence or next task for the group after any topic is discussed.

- **Proposal for a Scientific Benchmarking Website and "Manuals"**

  - Geoffrey Fox proposed creating a simple website resource to compile "manuals" or "handbooks" on different aspects of benchmarking, fitting the concept of "benchmark carpentry".

  - Gregor von Laszewski committed to setting up the web framework by the following week.

  - Gary Mazzaferro noted that the group's scientific benchmarking is more comprehensive than traditional benchmarking, leading to the decision to craft a new mission statement.

- **Lessons from the Paper Review Process and Strategy for Future Publications**

  - Feedback on an unsuccessful paper submission highlighted the need to communicate the difference between the group's scientific benchmark and others' ideas more carefully.

  - The strategy for future publications should focus on multiple short, separate papers on single areas (e.g., an energy benchmark paper) instead of long, comprehensive documents.

- **Strategy for Short, Focused Papers and Stakeholder Analysis**

  - Gary Mazzaferro proposed publishing multiple short (one- or two-page) papers as "internal notes" to document group activities, which Geoffrey Fox supported.

  - Benchmarking definitions and mission statements must clearly address the differing expectations of various stakeholders (e.g., technical vs. business analysts).

### **Presentation 1: Scaling Transformer Training in Stormer:**

- [Scaling Stormer Training\_ An Empirical Study of GPU Communication Overhead Optimization.pptx](https://docs.google.com/presentation/d/1JcEDMQfOk5HE0HbLBOZY2HuXb1x0jAeB/edit?usp=sharing&ouid=100475830980223085873&rtpof=true&sd=true)

<!-- -->

- **Stormer Model Overview and Features** Stormer is a medium-range weather forecasting model using a vision transformer backbone with weather-specific embedding to capture physical relationships between atmospheric variables (e.g., wind, pressure).

<!-- -->

- It uses a randomized iterative dynamic forecasting method, predicting changes over multiple time intervals (6-hour, 12-hour, 24-hour).

- It employs multi-step fine-tuning and a special pressure-weighted loss function that prioritizes variables near the surface.

- **Stormer Training Phases and Inference Strategies:** Training is divided into three phases, progressively increasing the number of consecutive predictions (rollout step method) to build stability for long forecasts.

<!-- -->

- Inference uses two strategies:

  - **Homogeneous:** Predicting the future using the same time intervals (e.g., four 6-hour steps for a 24-hour prediction).

  - **"Best \$m\$ in \$n\$":** Trying \$n\$ interval combinations, picking the \$m\$ best ones, and averaging the results for the final forecast.

- **Replication vs. Actual Stormer Differences:**

  - **Image Resolution:** Replication used 240x120; actual Stormer used 128x256.

  - **Data Split:** Replication used half the data (2000-2018); actual Stormer used 1979-2018.

  - **Patch Size:** Replication used 8; actual Stormer used 2 (larger size was necessary to avoid Out-of-Memory errors).

  - **Training GPUs:** Replication used 64 A100 GPUs; actual Stormer used 128 A100 GPUs.

  - **Experimental Software:** Setup included PyTorch 2.9.0, CUDA 12.8, nickel 2.27.5, and Distributed Data Parallel (DDP) on the NERSC Perlmutter supercomputer.

- **Stormer Scaling Behavior Analysis**

<!-- -->

- The study showed that with one node (four GPUs), training was computationally dominated (nickel all-reduce took 11% of GPU time).

- As the number of nodes increased, the training shifted to being communication dominated, consuming almost 87% of GPU time at 16 nodes (64 GPUs), indicating poor scaling.

- The communication dominance was due to increased average CUDA stream synchronization time.

- **Optimization Knobs Explored:**

  - **Mixed Precision (BF16):** Marginally reduced communication time from 77.8% to 77%.

  - **Gradient Bucket Size:** Increasing from 25 MB to 50 MB (and 100 MB) had a slight impact, reducing communication time to around 75.5-75.6%.

  - **Gradient Accumulation:** Increasing the factor significantly reduced communication time from 86% (factor of 1) to 28.7% (factor of 16).

- **Conclusion on Knobs:** Gradient accumulation was the most crucial factor in reducing communication overhead.

**Discussion on Compute vs. Communication Bottleneck**

- Matt Sinclair suggested the communication dominance might be due to insufficient compute to hide the communication overhead.

### **Victor Lu’s Presentation**

The document **"Data aspect of Scientific AI Benchmarks"** [Data aspect of Scientific AI Benchmarks](https://docs.google.com/document/d/14U8X6ZyKRuKpVS_15fp8HOSTBna0mdoIhxmuco16CaM/edit?tab=t.0#heading=h.trdg1lq3kk2o) discusses the critical role of data in interdisciplinary scientific AI and the challenges related to its management and benchmarking.

\

**Key points:**

- **Interdisciplinary Challenges:** Scientific breakthroughs increasingly rely on interdisciplinary work, but this is hampered by diverse domain-specific data formats, inconsistent standards, and infrastructure heterogeneity.

- **Data Formats:** Scientific research utilizes a wide array of specialized data formats, categorized by domain, including General/Multi-disciplinary (e.g., HDF5, NetCDF), Biology/Genomics (e.g., FASTA, BAM), Earth Sciences (e.g., GeoTIFF, Shapefile), and others. Key properties of these formats include precision, interoperability, scalability, and domain specialization.

- **Benchmarking Misalignment:** Traditional benchmarks, often focused on metrics like FLOPs or raw throughput, are insufficient for interdisciplinary workloads and often fail to account for domain-specific data characteristics, leading to suboptimal system design.

- **Database Benchmarking Lessons:** The history of database benchmarking (e.g., TPC-C, YCSB) shows a struggle against vendor manipulation (e.g., Oracle's "discrete transaction," Nvidia's TPCx-BB modifications) and challenges in cloud environments (data persistence, consistency).

- **Avoiding Benchmark Gaming:** To ensure fair and meaningful evaluation, the document advocates for adopting "evaluatological principles," including:

  - Designing benchmarks for **real-world relevance**.

  - Using **transparent assumptions** and a **suite of diverse, multi-dimensional metrics**.

  - Ensuring **independent verification** and **reproducible** open benchmarking.

  - Enacting **ex ante benchmark rules** that prohibit features with no practical value.

  - Continuously **reflecting and revising** the benchmark process.

The data aspect, through systematic frameworks for metadata, provenance, and schema harmonization, is positioned as the key enabler for collaborative, high-impact research.
